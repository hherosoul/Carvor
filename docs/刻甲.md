## 产品概述

刻甲（Carvor）是一款面向科研工作者的 AI 研究助手。它围绕科研工作的核心链路——**文献跟踪 → idea 锤炼 → 方法讨论 → 实验设计 → 论文写作**——提供端到端的智能辅助，同时通过内置的自我进化机制，越用越懂你。

科研人员最大的敌人不是不够聪明，而是被重复性劳动淹没：每天涌入的新论文看不完、实验配置反复试错、论文润色耗费心神、每次实验的经验无法沉淀。刻甲不替代你思考，而是替你干掉这些噪音，让你把精力集中在真正的创新上。

刻甲的功能分为两层：**全局层**包括文献跟踪与研究时间线（像雷达一样持续扫描领域动态）和 idea 锤炼（思维工作台，打磨研究想法）；**任务层**的研究任务，为每一个正式立项的研究课题提供独立的引用管理、综述生成、方法讨论、实验设计、论文润色环境，不同任务互不干扰。当 idea 锤炼成熟后，可以一键创建新任务，idea 文档和参考论文自动复制继承。

**技术架构**：Web GUI 前端 + 本地后端。数据存储在本地 SQLite + 文件系统，LLM 使用用户自定义的 OpenAI 兼容接口。文献数据通过 LLM 联网搜索获取（详见 2.1.2）。支持离线模式——仅 LLM 联网搜索获取新论文需要网络连接。

**核心差异化**：本地优先的隐私保护 + 持续自我进化的个性化 + 实验文档生成的工程化能力 + idea 锤炼到任务立项的科研链路闭环，是刻甲区别于任何单一工具的根本。

**LLM 优先设计原则**：刻甲采用"能交给 LLM 就交给 LLM"的设计哲学。凡是需要理解、判断、生成、分析的工作，均由 LLM 完成；系统只负责确定性的流程编排（数据存储、调度触发）。用户是最后一道关——LLM 的输出最终由用户审阅确认，因此不需要对 LLM 可能的不完美做过多的工程防御。刻甲的设计面向 LLM 能力的未来——随着大模型的快速进化，基于缺陷的设计会过时并成为系统的拖累，而刻甲的架构将随 LLM 进步而自然变强。遇到不确定的场景时，统一采用**"大模型决策 + 人类 prompt 约束"**的模式：信任大模型的判断能力，同时通过 prompt 中的明确指令约束底线。唯一需要严格约束的是**歪曲原意**和**臆造事实**，这通过 prompt 中的明确指令来防范。

## 0. LLM 调用约束原则

刻甲将大量工作交给 LLM，但有两个底线绝不妥协：**不歪曲原意**、**不臆造事实**。以下约束适用于所有 LLM 调用场景，必须在 prompt 设计中严格执行。

### 0.1 防歪曲：保持原文语义完整性

| 场景 | prompt 中的强制指令 |
|------|---------------------|
| 论文润色 | "严格保持原文语义，不得改变任何论点、结论或数据含义" |
| 论文摘要生成 | "摘要必须严格基于论文原文内容生成，不得添加论文未提及的信息或推断" |
| 深度阅读总结 | "总结中的每个论点必须在原文或对话中有对应依据，不得推断或补充" |
| 综述生成 | "对每篇论文的描述必须忠于原文，不得夸大或缩小其贡献" |
| idea 锤炼 | "对比已有工作时，必须准确描述其内容，不得为突显新颖性而歪曲他人工作" |

### 0.2 防臆造：不捏造不存在的信息

| 场景 | prompt 中的强制指令 |
|------|---------------------|
| BibTeX 生成 | "BibTeX 条目中，无法确认的字段留空，不得编造或猜测" |
| 实验数据分析 | "分析结论必须基于提供的日志数据，数据不足时明确说明'数据不足以判断'，而非猜测" |
| 联网搜索论文 | "返回的论文必须真实存在，必须是最新的，不得编造论文标题、作者或摘要" |

### 0.3 通用 prompt 约束模板

所有 LLM 调用的 prompt 末尾，应附加以下通用约束（可根据场景调整措辞）：

```
约束要求：
1. 严格忠于原始内容，不得歪曲、夸大或缩小
2. 不确定的信息不要编造，宁可不写也不要瞎写
3. 如实标注你的不确定之处（如"此关联性较弱，仅供参考"）
4. 优先保证准确性，其次才是完整性
```

### 0.4 Skill 作为 prompt 上下文

每次 LLM 调用时，系统自动将与当前任务相关的 Skill 内容作为 prompt 上下文注入：
- 文献跟踪相关任务 → 注入文献跟踪 Skill
- idea 锤炼 → 注入 idea 锤炼 Skill
- 提示词文档生成 → 注入代码提示词生成 Skill
- 论文润色 → 注入论文润色 Skill

### 0.5 提示词层级与上下文压缩

刻甲的每次 LLM 调用，prompt 由多种来源拼装而成。随着对话推进，上下文会不断增长。当接近模型最大上下文窗口时，需要通过压缩机制释放空间，同时确保关键信息不丢失。**上下文压缩适用于所有 LLM 调用场景**——无论是流式对话还是异步处理，只要上下文接近阈值，就触发压缩。

#### 0.5.1 提示词层级

prompt 按重要性和可压缩性分为四个层级，从高到低：

| 层级 | 内容 | 压缩保护 |
|------|------|----------|
| **系统级** | 防歪曲约束（0.1）、防臆造约束（0.2）、通用约束模板（0.3） | **不可压缩** |
| **任务级** | 当前任务相关的 Skill 内容（0.4） | **不可压缩** |
| **领域关键** | 当前研究领域的核心定义、关键假设、核心结论 | **不可压缩**（由 LLM 根据 Skill 中的领域规则判断） |
| **可压缩** | 对话历史、中间结果、辅助性细节 | **可压缩**（由 LLM 根据 Skill 中的领域规则进行总结压缩） |

#### 0.5.2 基于领域知识的压缩机制

**领域规则的来源**：领域规则由用户建立并维护在 Skill 中。用户在首次使用或需要时，手动编辑 Skill 中的"领域规则"章节，定义本领域的核心概念、关键术语、重要假设和推理基石。

| 场景 | 领域规则来源 |
|------|-------------|
| 文献跟踪 | 文献跟踪 Skill 中的领域规则 |
| 研究任务 | 代码提示词生成 Skill 中的领域规则 |
| idea 锤炼 | idea 锤炼 Skill 中的领域规则 |
| 论文润色 | 论文润色 Skill 中的领域规则 |

**压缩流程**：

```
上下文长度达到 80% 阈值
       ↓
锁定不可压缩层：系统级约束 + Skill + 领域关键信息
       ↓
LLM 根据 Skill 中的领域规则对可压缩层标注优先级：
  - 匹配领域规则中的核心概念/关键术语 → 保留原文（提升为领域关键层）
  - 与领域相关但未命中核心规则 → 压缩为摘要
  - 与领域无关 → 高度概括
       ↓
LLM 对可压缩内容执行总结压缩 → 替换原始可压缩层
       ↓
继续对话
```

#### 0.5.3 压缩触发与保护规则

**触发条件**：当上下文长度达到模型最大上下文窗口的 **80%** 时，自动触发。适用于所有 LLM 调用场景（流式和异步）。

**上下文不变性原则**：对话进行中，系统级约束和 Skill 内容不可变更。唯一允许修改上下文的时机是上下文压缩。Skill 更新、约束调整等操作在对话结束后生效。这一原则确保：(1) LLM 行为在对话内保持一致；(2) 若底层 LLM 服务支持 prompt caching，不会因中途修改系统级内容而破坏缓存导致成本飙升。

**压缩保护规则**：遵循 0.5.1 层级表的压缩保护定义——系统级、任务级（Skill）、领域关键信息不可压缩；仅可压缩层（对话历史、中间结果）可被总结概括，但需保留核心语义。

**用户感知**：压缩发生时，界面显示轻量提示（如"上下文已压缩"），让用户理解后续对话中 LLM 可能不记得早期对话的完整细节。

## 1. 产品定位与目标用户

刻甲（Carvor）是一个面向博士、博士后及科研工作者的智能研究辅助工具——不直接替代创新思考，而是通过自动化信息跟踪、结构化实验设计文档生成、高质量论文润色以及持续自我进化，将研究者从繁琐的重复性劳动中解放出来。

**核心用户**：博士研究生（尤其是高年级）、博士后、科研工作者（包括计算生物学、生物信息学、AI for Science 等领域）。扩展用户为其他需要编程和文献工作的科研领域。

**核心痛点**：文献跟进困难（难以持续跟踪最新成果，手工管理引用耗时）+ 实验调试低效（反复试错浪费资源）+ 论文写作繁琐（润色和术语统一消耗精力）+ 知识无法沉淀（经验教训无法系统化复用）。

### 1.2 竞品分析与差异化

现有工具各有侧重但均未闭环：Semantic Scholar / Consensus 仅搜索不跟踪不沉淀；Elicit / SciSpace 无实验设计和自我进化；ChatGPT / Claude 无领域知识沉淀和文献跟踪；ReadPaper 仅覆盖阅读环节。

**刻甲的核心差异化**：全链路闭环（文献发现→idea锤炼→方法讨论→实验设计→代码生成→数据分析→论文润色）+ 自我进化（持续积累用户偏好和领域知识）+ 本地优先（数据存本地，隐私有保障）+ 实验工程化（生成结构化提示词文档供编程助手使用，现有竞品均未覆盖）。

## 2. 核心功能模块

### 2.1 文献跟踪与研究时间线

**目标**：让用户始终站在领域前沿——不只是"看到新论文"，而是像一位专注的科研助手帮你盯着动态、整理脉络，形成清晰的研究进展时间线。

#### 2.1.1 论文库与领域描述

每个论文库由用户的研究领域描述定义。领域描述是一段自然语言，描述用户的研究方向和兴趣——不是关键词，而是对自身研究领域的完整表达。

- **领域描述**：用户用自然语言描述自己的研究领域（如"我研究基于扩散模型的蛋白质结构预测，关注扩散模型在分子生成中的新变体、等变神经网络架构的改进，以及蛋白质设计中的评估方法"）。LLM 根据这段描述联网搜索最新论文，获取领域及强相关领域的最新成果。
- **领域描述的个性化**：一个优秀的科研工作者必然专注于某一领域。如果他的研究方向发生变化，时间线上的论文信息会自然揭示这一转变——研究路线的发展历史一目了然，更加体现用户的个性化。
- **领域描述无需 LLM 判断**：研究领域由研究者自行描述，不需要 LLM 来判断或推断用户的研究领域。

> **说明**：idea 锤炼和 Skill 进化知识是**全局的**，与论文库无关。无论用户切换到哪个论文库，idea 列表和 Skill 均保持一致。领域描述仅仅是论文搜索的依据。论文内容是否有关联、是否有价值，完全由科研人员自行判断。

**语义搜索**：论文库支持基于向量嵌入的语义搜索，用户可用自然语言查询论文（如"用CNN做特征学习的DTA研究"）。系统使用本地部署的 BAAI/bge-small-zh-v1.5 嵌入模型（存储在 `backend/models/` 目录），通过 LlamaIndex 构建向量索引，异步执行搜索不阻塞主进程。向量搜索无结果时自动回退到 LLM 搜索和关键词搜索。

#### 2.1.2 LLM 联网搜索配置

**文献数据来源**：刻甲通过 LLM 的联网搜索能力获取文献数据。LLM 根据用户的领域描述，联网搜索最新论文，以结构化 JSON 输出，系统解析后存入 SQLite。

**LLM 配置**：用户可自定义 LLM 服务，系统支持所有 OpenAI 兼容接口（包括国内主流大模型）：
- API Base URL、API Key、模型名称
- **联网搜索工具配置**：LLM 需具备联网搜索能力（如 web_search 工具），系统在调用时自动传入搜索工具名称
- 系统在前端展示 LLM 的 Token 使用情况，帮助用户控制成本
- **配置版本化**：LLM 配置文件包含版本号，应用升级时自动迁移旧配置，确保向前兼容

**LLM 调用模式**：异步模式（每日搜索、周报、进化学习——后台处理，前端显示进度）和流式模式（深度阅读、idea锤炼、润色、提示词文档——实时交互，逐步输出）。

**LLM 调用重试策略**：
- LLM 处理失败：自动重试，**最多 3 次**，超过 3 次则标记失败并通知用户

**本地论文导入**：除了通过 LLM 联网搜索获取文献，用户还可通过 Web GUI 导入本地论文 PDF。导入后系统执行以下处理：
1. 将 PDF 存储到论文库文件夹
2. **LLM 提取论文元数据**：标题、作者、作者院校、摘要等
3. 将元数据存入 SQLite 索引（去重规则：标题相同则判定重复，不插入）
4. 将论文加入时间线（标记为"手动导入"）

#### 2.1.3 研究时间线

文献跟踪的核心界面——采用**两级导航**的时间线设计。

**第一级：时间线总览**

主时间线以周为节点纵向排列，每个节点展示该周的概要信息。

**周节点卡片内容**：

| 元素 | 说明 |
|------|------|
| 时间范围 | 该周的起止日期（如"4.14 — 4.20"） |
| 论文数量 | 当周收录论文总数 |
| 感兴趣标记 | 用户已标记"感兴趣"的论文数量（如有则高亮显示） |
| 未读状态 | 若该周有用户未查看的新论文，节点显示未读标记 |

> **说明**：LLM 只负责提供准确的、完整的、科学的总结，不负责判断"亮点"——是否为亮点由科研人员自行判断。

**第二级：周详情页**

用户点击某个周节点后，进入该周的论文详情页。

**论文详情页布局**：
- **顶部**：显示该周时间范围和论文总数，提供按来源的筛选栏
- **论文列表**：按时间倒序排列所有论文卡片
- **感兴趣筛选**：可一键切换为仅显示用户标记为"感兴趣"的论文

**论文卡片内容**：

| 元素 | 说明 |
|------|------|
| 标题 | 论文标题 |
| 作者及院校 | 作者姓名及所属院校 |
| 来源 | LLM搜索 / 手动导入 |
| LLM 摘要 | 由 LLM 生成的结构化摘要（≤200字，非一句话） |
| 已读状态 | 论文库级别的已读/未读标记 |
| 操作按钮 | "标记感兴趣/取消感兴趣""下载全文""深度阅读""加入任务引用池""写笔记""标记已读""删除" |

**已读状态**：已读/未读状态是**论文库级别**的——同一篇论文在不同论文库中可以有不同的已读状态。

**感兴趣标记**：为切换操作，再次点击可取消感兴趣状态。

**引用池去重**：同一论文不可重复加入同一任务的引用池，重复添加时系统提示"已在引用池中"。

**阅读笔记**：用户可在论文卡片上点击"写笔记"记录阅读灵感，笔记按时间排列，可在左侧菜单"阅读笔记"中统一管理（详见 2.1.7）。

**进度提示**：搜索显示"正在搜索…"，总结显示"正在总结…"，周报显示 **12/30** 格式进度。

#### 2.1.4 自动化跟踪与每日搜索

**每日自动搜索**：系统每日定时触发 LLM 联网搜索，使用论文库配置的领域描述进行检索。

**搜索策略**：**错过好过泛滥**。LLM 负责筛选，只保留与领域描述高度相关且质量较高的论文。LLM 必须保证返回的论文**真实存在**且**具有时效性**——必须是最新论文，不得编造。

**搜索与处理流程**（异步）：

```
1. 读取论文库的领域描述
       ↓
2. LLM 联网搜索（传入领域描述 + web_search 工具）
   LLM 搜索领域及强相关领域的最新论文
   若 LLM 判断结果不足，可发起补充搜索（最多3轮迭代）
       ↓
3. LLM 以结构化 JSON 输出论文列表：
   - 每篇论文包含：标题、作者、院校、摘要、发表日期、来源URL
   - 筛选与领域描述高度相关的论文（错过好过泛滥）
   - 论文必须真实存在，必须是最新的
       ↓
4. 解析 LLM 输出 JSON → 写入 SQLite 数据库
   去重规则：URL 和标题均相同则判定重复，不插入
   注：此时不下载论文全文
       ↓
5. 用户在时间线上看到新论文卡片
```

**周报（按需生成）**：

界面上展示各周的周报列表。用户点击某一周的周报时，系统启动异步生成流程：

- **周报上限**：每周最多 **30 篇**论文
- **摘要要求**：每篇论文的摘要**最多 200 字**，且不能是一句话——必须是结构化的多句摘要
- **周报内容**：LLM 只负责提供准确的、完整的、科学的总结，不负责判断"亮点"
- **生成方式**：异步处理，前端显示生成进度（按 **12/30** 格式显示进度）
- **报告格式**：Markdown

**按需查询**：用户主动询问时（如"最近两周在蛋白质设计领域有哪些突破？"），LLM 联网搜索指定时间范围的论文，处理后生成定制化报告（异步）。

#### 2.1.5 论文全文下载与深度阅读

**论文全文下载**（用户主动触发）：

用户在时间线中点击论文的"下载全文"按钮（支持批量下载），系统尝试获取 PDF：

1. **下载 PDF**：LLM 提供的来源 URL 中若包含可直接下载的 PDF 链接，则自动下载；否则提示用户该论文需手动下载。
2. **存储**：下载的 PDF 存储到论文库文件夹。
3. 下载完成后，论文卡片上的"深度阅读"按钮变为可用状态。

> **设计理念**：随着大模型能力进化，越来越多的论文将能直接点击下载。当前无法自动下载的，用户手动下载即可——这与"和大模型一起进化"的理念一致。

**深度阅读**（前提：论文全文已在论文库中）：

用户点击"深度阅读"后，界面进入论文阅读模式——中间区域展示论文全文，右侧打开与 LLM 对话的界面。

1. **界面布局**：进入三栏模式——左侧导航栏（可折叠）、中间论文全文区、右侧 LLM 对话区（可折叠）。
2. **对话式阅读**：用户在右侧对话区与 LLM 就当前论文进行讨论。LLM 基于论文全文内容回答用户的问题。
3. **对话总结**：用户结束深度阅读时，需手动确认是否生成总结。若用户认为讨论有意义，确认后 LLM 将完整对话记录总结为深度阅读结论（≤1000字），存入 SQLite 中与该论文关联。同一篇论文只保留一份总结——每次确认总结时，LLM 基于全部对话记录重新生成，替换旧总结，而非增量追加。
4. 用户的思考和观点同样存储在 SQLite 中该论文的索引记录中。
5. 离线可用：已下载的论文深度阅读完全离线可用。
6. **对话持久化**：所有 LLM 对话（深度阅读、idea锤炼、综述讨论、方法讨论等）均持久化存储。刷新页面或重启应用后，可恢复最近一次对话会话，继续未完成的讨论。

> **说明**：深度阅读是研究的前置工作，与具体任务无关——阅读结果存入论文库，任何任务都可引用。

#### 2.1.6 阅读笔记

**定位**：全局的论文阅读灵感记录工具——用户在阅读论文时随时记录灵感和想法，不绑定任何具体任务。

**功能说明**：

- **入口**：左侧菜单"阅读笔记"，以及论文卡片上的"写笔记"按钮
- **记录内容**：笔记与论文关联，包含笔记内容和创建时间
- **笔记列表**：按时间倒序排列，显示创建时间、论文标题、笔记内容摘要
- **操作**：查看笔记详情、删除笔记
- **数据存储**：SQLite `paper_notes` 表（id, paper_id, content, created_at）

> **说明**：阅读笔记是轻量级的灵感记录，与深度阅读的对话总结不同——笔记是用户主动记录的片段式想法，深度阅读总结是 LLM 对完整对话的概括。

#### 2.1.7 idea 锤炼

**定位**：全局的"思维工作台"——用户在这里打磨研究想法，不绑定任何具体任务。

idea 本质上是一个 Markdown 文档，全局唯一，不与任何论文库绑定。

**功能流程**（流式 LLM 交互）：

1. **idea 提交**：用户用自然语言描述一个初步想法。
2. **Agent 结构化分析**：
   - **新颖性评估**：LLM 联网搜索相关论文，评估 idea 与已有工作的差异点。查重完全基于 LLM 能力，不依赖本地论文库。
   - **假设梳理**：提取 idea 中隐含的关键假设，逐条评估是否有依据。
   - **可行性分析**：评估技术可行性（数据是否可获得、计算资源是否够、现有方法是否可复用）。
   - **风险识别**：标注潜在风险点。
3. **多轮对话打磨**：用户和 Agent 反复讨论，Agent 追问模糊之处，帮助用户把粗糙的想法变成可执行的研究方案。
4. **参考论文**：锤炼过程中，LLM 联网搜索相关论文，用户标记为"对 idea 有参考价值"的论文会被记住。
5. **输出**：结构化的 idea 文档（Markdown），包含以下章节：

| 章节 | 说明 |
|------|------|
| 最终版 idea | 锤炼完成后的研究想法描述 |
| 关键假设 | idea 中隐含的核心假设列表 |
| 预期实验方向 | 基于假设推导的实验方向 |
| 风险评估 | 已识别的潜在风险 |
| 参考论文列表 | 标记为有参考价值的论文 |
| 演化记录 | idea 从初版到最终版的修改历史 |

**idea 的生命周期**：

```
萌芽（用户随手记一个想法）
  → 锤炼（结构化分析 + 多轮对话）
    → 决策点：
      ├── 放弃（归入"已放弃"列表，保留供回看）
      ├── 暂停（留在"待继续"列表）
      └── 立项（创建新任务，详见 2.2.1）
```

**全局 idea 列表**：用户可查看所有 idea 的状态（锤炼中 / 已放弃 / 已暂停 / 已立项）。

> **说明**：idea 锤炼是全局功能，不绑定任何论文库。成熟后可选择"创建任务"落地，详见 2.2 节。

### 2.2 研究任务

**目标**：为每一个正式立项的研究课题提供独立的工作环境，将引用管理、综述生成、方法讨论、实验代码、论文润色等任务级功能隔离在不同的任务中，互不干扰。

**核心思想**：文献跟踪和 idea 锤炼是"全局雷达"，扫描和筛选；而引用、综述、实验、润色都是"干活"，必须绑定在具体的研究任务上。

**任务流程**：立项后，任务详情页以 **Steps 进度条**展示7步流程，各步有顺序关系但可独立调整：

```
引用池 → 综述讨论 → 方法讨论 → 研究文档 → 提示词文档 → 实验 → 论文润色
```

- **引用池**：管理任务引用的论文，支持 BibTeX 生成
- **综述讨论**：用户与 LLM 讨论撰写文献综述，内容可编辑
- **方法讨论**：用户与 LLM 讨论梳理方法框架，内容可编辑
- **研究文档**：由综述和方法合并自动生成，需先完成前两步；生成后可手动编辑
- **提示词文档**：基于研究文档与 LLM 讨论生成实验提示词
- **实验**：上传训练日志，LLM 分析实验结果
- **论文润色**：用户与 LLM 讨论润色论文，内容可编辑

**所有内容区均可编辑**：综述、方法、研究文档、论文润色均支持编辑/阅读模式切换，用户可直接修改 Markdown 内容。

**从 idea 到任务**：当用户在 idea 锤炼中认为一个想法成熟了，可以一键"创建任务"。idea 文档**自动复制到任务工作空间**，原 idea 工作空间中的文档保留不变。参考论文自动导入任务引用池。

#### 2.2.1 任务管理

| 能力 | 说明 |
|------|------|
| 创建任务 | 唯一入口：从 idea 立项（idea 文档 + 参考论文自动导入引用池） |
| 任务列表 | 左侧导航展示所有任务，按月折叠展示 |
| 任务切换 | 点击任务名称即可切换，同一时间只有一个活跃任务 |
| 任务隔离 | 不同任务的数据互不可见、互不干扰 |
| 任务删除 | 支持用户手动删除任务（需反复确认），删除后数据不可恢复 |

**从 idea 创建任务时，自动继承**：

| 继承内容 | 来源 |
|----------|------|
| 任务名称 | idea 标题（用户可修改） |
| 研究目标 | idea 文档中的关键假设和预期方向 |
| 引用池初始论文 | idea 锤炼中标记为参考的论文 |
| idea 文档 | 完整的 idea 锤炼记录（自动复制，原 idea 保留） |
| 风险评估 | idea 锤炼中的风险识别 |

**任务的数据边界**：

| 数据类型 | 作用域 | 说明 |
|----------|--------|------|
| 引用池 | 任务级 | 每个任务独立的 `references.bib`，不可重复添加同一论文 |
| 研究文档 | 任务级 | 由综述和方法合并生成，可手动编辑 |
| 对话会话 | 任务级 | 持久化的 LLM 对话历史，按场景分类，刷新/重启后可恢复 |
| 提示词文档 | 任务级 | 实验配置和代码文档存储在任务内 |
| 实验日志与分析 | 任务级 | 训练日志、分析报告与任务关联 |
| 润色文档 | 任务级 | 论文写作内容属于当前任务 |
| 阅读笔记 | 全局 | 轻量级灵感记录，与论文关联，按时间排列 |
| 文献跟踪时间线 | 全局 | 所有任务共享 |
| idea 锤炼 | 全局 | 不绑定论文库 |
| 论文库 | 全局 | 本地文件夹 + SQLite 索引，任何任务可引用 |
| 自我进化知识 | 全局 | Skill 目录，跨任务、跨论文库共享 |

#### 2.2.2 引用管理

- **从时间线选入**：从时间线论文卡片点击"加入任务引用池"。
- **idea 继承**：从 idea 创建任务时，参考论文自动导入引用池。
- **手动添加**：支持粘贴 DOI、标题添加论文。LLM 联网搜索获取元数据。
- **BibTeX 生成**：由 LLM 生成 BibTeX 条目，比模板生成更准确。
- **标签分组**：引用池内的论文可打标签。LLM 根据论文内容和任务研究方向自动推荐标签。
- **跨任务不可见**：A 任务的引用池对 B 任务不可见。

#### 2.2.3 综述生成

**定位**：研究文档演化的第二阶段——从 idea 出发，通过与 LLM 讨论，基于引用池论文撰写综述，回答"别人做了什么"。

**核心流程**（流式 LLM 交互）：

1. **输入上下文**：idea 文档 + 引用池论文，作为 LLM 讨论的基础。
2. **多轮对话**：用户与 LLM 讨论综述结构和内容——LLM 基于引用池论文和 idea 中的研究方向，帮助用户梳理领域脉络、组织综述框架、撰写各部分内容。综述中对每篇论文的描述必须忠于原文。
3. **自动关联引用**：LLM 在综述中自动关联引用池中的 BibTeX 条目。
4. **迭代修改**：用户可反复修改，LLM 迭代完善。
5. **输出**：综述部分写入研究文档（Markdown + BibTeX）。

#### 2.2.4 方法讨论

**定位**：研究文档演化的第三阶段——基于综述中的已有工作，通过与 LLM 讨论，梳理方法框架，回答"我要怎么做"。

**核心流程**（流式 LLM 交互）：

1. **输入上下文**：研究文档（idea + 综述）+ 引用池论文，作为 LLM 讨论的基础。
2. **多轮对话**：用户与 LLM 讨论方法设计——LLM 基于综述中的已有工作和 idea 中的假设，帮助用户梳理方法框架、技术路线、创新点表述。
3. **迭代修改**：用户可反复修改，LLM 迭代完善。
4. **输出**：方法部分写入研究文档，至此研究文档完成三阶段演化（详见 2.2 概述）。
5. **衔接实验**：方法讨论成熟后，用户可基于方法内容直接进入提示词文档生成——方法中的技术路线自然成为实验设计的输入。

> **设计理念**：综述回答"别人做了什么"，方法讨论回答"我要怎么做"，提示词文档回答"怎么验证"。三者形成完整的科研逻辑链，统一承载在一份持续演化的研究文档中。

---

### 2.3 实验代码提示词文档生成

**目标**：在当前任务内，生成高质量的提示词文档，供编程助手无歧义地生成可执行实验代码，并通过数据分析反馈改进实验设计。

#### 2.3.1 提示词文档模板

采用 Markdown 格式，包含以下章节（关键部分需提供伪代码和示例）：

| 章节 | 必填内容 |
|------|----------|
| 项目名称 | 简短描述 |
| 目标概述 | 研究问题、预期成果 |
| 数据说明 | 来源、格式、规模、预处理步骤（含伪代码示例）、划分方式 |
| 模型架构要求 | 基础架构、输入输出尺寸、特殊约束（可提供参考论文或伪代码） |
| 训练配置 | 损失函数、优化器、学习率调度、批量大小、训练轮数、早停设置、硬件要求 |
| 超参数搜索策略 | 搜索方法、候选范围、评估指标 |
| 监控与日志 | 记录指标、频率、早停条件（含伪代码） |
| 输出要求 | 最佳模型路径、日志文件格式、图表文件命名及格式 |
| 依赖环境 | Python 版本、必需库、安装命令 |
| 验收标准 | 可勾选的检查清单 |

#### 2.3.2 交互式生成流程

1. 用户以自然语言描述实验想法（流式 LLM 交互）。LLM 的输入上下文包含当前任务的研究文档（论文雏形），确保实验设计与研究目标一致。
2. Agent 通过对话追问缺失信息（参照模板字段），逐步完善文档。
3. 用户可随时修改文档内容。
4. 最终输出完整的 Markdown 文件，用户可下载或直接复制发送给编程助手。

#### 2.3.3 实验数据分析与反馈

- 用户将训练日志（**JSON 格式**）和模型输出提供给 Agent。
- **LLM 分析数据**：训练是否收敛、超参数敏感性、可能的问题（过拟合、欠拟合等）。
- **LLM 生成分析报告**，给出改进建议。数据不足时明确说明"数据不足以判断"。
- **多轮实验对比**：LLM 对同一任务内的多次实验进行对比分析。
- **任务隔离**：实验记录和分析报告存储在当前任务内。

### 2.4 论文润色与写作辅助

**目标**：在当前任务内，提升论文语言质量，保持专业风格，节省润色时间。

#### 2.4.1 润色功能

- **语法与拼写修正**：LLM 自动纠正错误。
- **流畅性与表达优化**：LLM 重写啰嗦或不通顺的句子，保持原意。
- **期刊风格适配**：用户可选择目标期刊，LLM 调整语言风格。风格偏好存储在论文润色 Skill 中。
- **术语一致性检查**：术语映射表存储在论文润色 Skill 中，LLM 确保全文术语统一。
- **避免大模型常用词**：禁用词列表存储在论文润色 Skill 中，LLM 在润色时检测禁用词并提供自然替代。用户可通过 Skill 自行增删禁用词。

> **防歪曲约束**：论文润色是刻甲中 LLM 最需要严格约束的场景，必须遵循 0.1 节防歪曲约束，且额外增加"如果原文表述可能有歧义，优先保留原文而非猜测意图"、"不得添加原文未提及的新观点、新论据或新引用"。

> **说明**：论文润色环节中所有自定义配置（术语映射表、禁用词列表、风格偏好、期刊模板等）统一通过论文润色 Skill 管理。

#### 2.4.2 交互方式

- 用户上传 `.docx` 文件。
- **润色界面**：单文档视图，所有内容以 Markdown 为基准渲染。修改建议以**内联差异标注**呈现：删除的原文用红色删除线，新增或替换的内容用绿色标注。
- **操作控制**：文档顶部提供"保留"和"撤销"按钮，支持逐条或批量操作。
- 用户可逐条或一键全部接受/拒绝修改。
- 处理过程中显示明确的进度条反馈。

### 2.5 Agent 自我进化

**目标**：Agent 在使用过程中不断学习用户的偏好、实验规律、领域知识，变得越来越智能和个性化。所有进化知识都是**全局的**——进化的本质是"Agent 越来越懂你"。

#### 2.5.1 进化维度

**核心维度（5 个）**：

| 维度 | 说明 | 写入 Skill |
|------|------|-----------|
| 算法模型知识 | 记录用户所在领域常见的超参数范围、成功/失败模式 | 代码提示词生成 Skill |
| 提示词文档模板优化 | 根据用户对生成文档的修改和补充，优化模板结构 | 代码提示词生成 Skill |
| 论文质量评价 | 识别用户常引用的论文、点赞的观点、标记为"重要"的工作 | 文献跟踪 Skill |
| 个性化风格 | 学习用户的思考方式、问答偏好、对话习惯 | idea 锤炼 Skill |
| 实验策略优化 | 从多次实验中总结规律 | 代码提示词生成 Skill |

**扩展维度（2 个）**：

| 维度 | 说明 | 写入 Skill |
|------|------|-----------|
| 文献偏好建模 | 学习用户在时间线上的阅读偏好 | 文献跟踪 Skill |
| 写作风格记忆 | 学习用户的写作习惯 | 论文润色 Skill |

**学习方式**：

| 学习方式 | 适用场景 | 说明 |
|----------|----------|------|
| **LLM 实时交互观察** | idea 锤炼 | LLM 在对话过程中直接观察用户偏好 |
| **日志记录 + 定期 LLM 总结** | 引用、标记、编辑、实验等 | 系统将操作记录到日志，定期由 LLM 提取模式 |

#### 2.5.2 进化机制——三级递进管道

进化不是"学到就写"，而是一个渐进确认的过程，防止噪声和错误知识污染 Agent 的能力。

**三级递进流程**：

```
交互或日志产生学习信号
       ↓
┌──────────────────────┐
│  第1级：初级进化文档    │  LLM 自动记录原始观察
│  (observing.md)       │  "发现用户在蛋白质任务中总用 lr=0.001"
└──────────┬───────────┘
           │ LLM 判断 observing.md 中出现重复模式（语义匹配，非字符串匹配）
           ↓
┌──────────────────────┐
│  第2级：待进化文档      │  候选知识，等待验证
│  (candidate.md)       │  LLM 提示用户是否采纳
└──────────┬───────────┘
           │ 用户显式确认写入
           ↓
┌──────────────────────┐
│  第3级：SKILL.md       │  已验证知识，写入对应 Skill
│                       │  成为 Agent 可直接调用的能力
└──────────────────────┘
```

**各级规则**：

| 级别 | 文档 | 写入条件 | 用户参与 |
|------|------|----------|----------|
| 第1级 | `observing.md` | LLM 自动从交互中提取模式 | 无需确认 |
| 第2级 | `candidate.md` | LLM 判断 observing.md 中出现重复模式 | LLM 提示用户是否采纳 |
| 第3级 | `SKILL.md` | 待进化条目由用户显式确认写入 | 用户确认写入 |

**Skill 防膨胀**：每次对 Skill 进行进化时，LLM 必须对整个 Skill 文档进行总结压缩，保持 Skill 内容在**约 800 字**以内。

**进化回滚**：任何已写入 SKILL.md 的知识，用户可在进化日志中标记为"有误"，回退到待进化文档重新验证。

#### 2.5.3 Skill 体系

进化知识的最终载体是 Skill，存储在项目的 `skills/` 目录下。每个 Skill 是一个以其名字命名的文件夹，核心文件为 `SKILL.md`。

**四类 Skill**：

| Skill 类别 | 对应功能模块 | 覆盖进化维度 | 用户可预置配置 |
|------------|-------------|-------------|---------------|
| **文献跟踪** | 2.1 文献跟踪与研究时间线 | 论文质量评价、文献偏好建模 | 无 |
| **idea 锤炼** | 2.1.6 idea 锤炼 | 个性化风格 | 无 |
| **代码提示词生成** | 2.3 实验代码提示词文档 | 算法模型知识、提示词文档模板优化、实验策略优化 | 无 |
| **论文润色** | 2.4 论文润色与写作辅助 | 写作风格记忆 | 禁用词列表、术语映射表、期刊风格偏好 |

**SKILL.md 内部结构**和**Skill 文件存储结构**详见架构设计文档 4.3 节。

### 3.4 其他要求

- 支持深色/浅色主题切换。
- 响应式设计，适配不同屏幕尺寸。
- **本地优先**：论文全文存储在本地论文库文件夹，索引数据存储在 SQLite 中，不上传云端。
- 中英文双语，随系统语言切换。
- **支持离线模式**：详见第 4 节非功能需求。
- **删除操作需严格确认**：无法恢复的删除操作，需弹出确认框，用户必须**输入一段随机显示的文字**才能确认。系统无数据备份与恢复机制。
- **所有组件均为开源软件**。

## 7. 已知风险与约束

| 风险 | 缓解措施 |
|------|----------|
| LLM 联网搜索可能返回不存在的论文 | prompt 中强制约束"返回的论文必须真实存在，必须是最新的"；用户是最后一道关，审阅时可发现 |
| LLM 服务不可用 | 自动重试最多 3 次；超过重试次数标记失败并通知用户；支持离线模式 |
| 长文本 PDF 解析失败 | 允许用户手动复制关键段落 |
| 编程助手生成的代码可能仍有错误 | 在提示词文档中加入验收标准；鼓励用户在 IDE 中迭代调试 |
| **论文润色歪曲原意** | 内联差异标注让用户逐条审阅；prompt 中强制要求"严格保持原文语义" |
| **自我进化知识污染** | 三级递进管道严格执行；Skill 防膨胀机制（≤800字）；用户可手动清除和回退 |
| 无数据备份机制 | 所有无法恢复的删除操作必须输入随机文字确认 |

> **关于 LLM 输出质量的说明**：刻甲遵循"用户是最后一道关"的原则。LLM 生成的内容可能不完美，但用户会自行检查。不需要为此做过多的工程防御。唯一需要严格约束的是**歪曲原意**和**臆造事实**——通过 prompt 中的明确指令来防范。

## 8. 术语表

| 术语 | 定义 |
|------|------|
| Agent | 刻甲的智能助手核心，基于大语言模型，能理解自然语言指令、执行任务、持续学习 |
| 研究任务 | 为一个研究课题提供的独立工作环境，包含引用管理、综述生成、方法讨论、实验设计、论文润色等功能，不同任务互不干扰 |
| 研究文档 | 任务内持续演化的文档：idea → +综述 → +方法 = 论文雏形，三个阶段均为与LLM讨论式生成 |
| 对话会话 | 持久化的 LLM 对话历史，按场景（深度阅读、idea锤炼、综述讨论、方法讨论等）分类存储，刷新/重启后可恢复 |
| 上下文不变性 | 对话进行中系统级约束和 Skill 内容不可变更的原则，唯一例外是上下文压缩 |
| idea 锤炼 | 全局的"思维工作台"，用户在这里打磨研究想法，成熟后可创建新任务 |
| idea 文档 | 结构化的 Markdown 文档，包含：最终版 idea、关键假设、预期实验方向、风险评估、参考论文列表、演化记录 |
| 提示词文档 | 一种结构化的 Markdown 文档，描述实验的完整配置，供编程助手无歧义地生成实验代码 |
| 方法讨论 | 综述完成后的方法设计环节，用户与 LLM 多轮对话产出论文方法部分初稿，衔接综述与实验设计 |
| 研究时间线 | 文献跟踪的核心界面，采用两级导航：第一级为按周排列的概要总览，第二级为周详情页（论文卡片列表，含 LLM 生成的结构化摘要≤200字） |
| 领域描述 | 用户对自身研究领域的自然语言描述，定义论文库的搜索范围。LLM 根据领域描述联网搜索最新论文 |
| 论文库 | 按领域描述分类的论文存储，包含论文全文（本地文件夹）和索引数据（SQLite）。通过顶部 Tab 切换 |
| 引用池 | 任务内的论文引用集合，存储为 BibTeX 格式，与论文库为引用关系而非复制关系 |
| Skill | Agent 进化知识的最终载体，存储在 `skills/` 目录下，每个 Skill 是以其名字命名的文件夹，核心文件为 `SKILL.md`。也是用户自定义配置的入口 |
| 三级递进管道 | 进化知识的渐进确认机制：observing.md → candidate.md → SKILL.md。由 LLM 驱动，Skill 每次进化后总结压缩至约 800 字 |
| 编程助手 | 用户实际编写代码的工具（如 Trae、Cursor、Claude Code），刻甲生成的提示词文档供其使用 |
