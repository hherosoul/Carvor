## 0. 架构核心哲学

### 0.1 "简单固定流程 + 聪明LLM"

刻甲的架构遵循一条铁律：**系统只做确定性的管道编排，所有需要理解、判断、生成的环节全部交给LLM。**

| 系统负责 | LLM负责 |
|----------|---------|
| 数据库读写、文件管理 | 论文搜索、筛选|
| 定时调度、任务队列管理 | 摘要生成、综述撰写、论文润色 |
| UI渲染、WebSocket推送 | idea新颖性评估、假设梳理、风险识别 |
| Prompt拼装、约束注入、Skill注入 | 进化信号提取、重复模式识别、知识压缩 |
| JSON解析、数据存储 | 上下文压缩中的优先级标注与语义压缩 |

**系统不判断**——不判断论文是否相关、不判断idea是否新颖、不判断润色是否到位。

**系统只搬运**——数据存储、文件管理、UI渲染、调度触发。

**LLM做决策**——搜索、筛选、摘要、分析、生成、压缩、进化判断。

### 0.2 系统与LLM的通信契约

系统与LLM之间通过**结构化JSON Schema**通信。系统喂给LLM结构化输入，LLM吐回结构化输出，系统再执行确定性的存储、调度、渲染。系统不解析LLM的自然语言输出做业务决策——只有面向用户的场景（对话、润色、综述）才输出自然语言。

### 0.3 LLM Gateway是唯一出口

所有LLM调用必须经过LLM Gateway，无一例外。Gateway确保：

1. **约束不遗漏**——每次调用自动注入防歪曲约束、防臆造约束、通用约束模板
2. **Skill不遗漏**——根据场景自动注入对应Skill
3. **上下文不溢出**——检测上下文长度，超80%阈值自动触发压缩（适用于所有LLM调用场景）
4. **失败不丢失**——自动重试最多3次，超过则标记失败并通知用户
5. **联网搜索工具注入**——根据场景需要，自动传入web_search工具名称
6. **上下文不变性**——对话进行中，系统级约束和Skill内容不可变更。唯一允许修改上下文的时机是上下文压缩。Skill更新、约束调整等操作在对话结束后生效

## 2. LLM Gateway 设计

LLM Gateway是整个架构的心脏。所有LLM调用必须经过Gateway，确保PRD第0节的约束在每个调用中被严格执行。

### 2.1 Gateway处理流程

```
业务Pipeline发起LLM调用请求
        │
        │  请求结构：
        │  {
        │    scenario: "paper_search",     // 调用场景标识
        │    input_data: {...},            // 业务数据
        │    output_schema: {...},         // 期望的输出JSON Schema
        │    mode: "async" | "stream",    // 调用模式
        │    context: [...],              // 对话上下文（流式模式）
        │    skill_name: "文献跟踪",       // 需要注入的Skill
        │    enable_web_search: true,     // 是否启用联网搜索
        │    web_search_tool_name: "$web_search"  // 联网搜索工具名称
        │  }
        │
        ▼
┌─────────────────────────────────────┐
│          LLM Gateway                │
│                                     │
│  Step 1: Prompt组装                 │
│  ├─ 拼装系统级约束（0.1-0.3）       │  ← system prompt（始终不变，可缓存）
│  ├─ 注入Skill上下文（0.4）          │  ← 用户消息（首条），根据scenario选择Skill
│  ├─ 拼装业务输入数据                │  ← 用户消息（首条），Pipeline提供的结构化输入
│  ├─ 拼装输出格式要求                │  ← 用户消息（首条），output_schema转为prompt指令
│  ├─ 注入联网搜索工具（如需）        │  ← enable_web_search时注入
│  └─ 检查上下文长度                  │  ← 超80%触发压缩Pipeline
│                                     │
│  Step 2: 调用执行                   │
│  ├─ 选择模式（异步/流式）           │
│  ├─ 发送请求到用户配置的LLM服务     │
│  └─ 失败重试（最多3次）             │
│                                     │
│  Step 3: 输出处理                   │
│  ├─ 异步模式：解析LLM返回的JSON     │
│  ├─ 流式模式：逐chunk推送到前端     │
│  ├─ Token计量记录（按scenario统计） │
│  └─ 返回结果给Pipeline             │
└─────────────────────────────────────┘
```

### 2.2 Prompt组装规则

Prompt按四层层级拼装。**Skill内容以用户消息注入而非system prompt**，原因：system prompt变化会破坏LLM服务的prompt caching，导致成本飙升；Skill进化会定期更新Skill内容，若注入在system prompt中，每次进化都会使后续所有对话缓存失效。

| 拼装顺序 | 层级 | 内容 | 注入位置 | 可压缩性 |
|----------|------|------|----------|----------|
| 1 | 系统级 | 防歪曲约束 + 防臆造约束 + 通用约束模板 | system prompt | 不可压缩 |
| 2 | 任务级 | 当前场景对应的Skill全文 | **用户消息**（首条） | 不可压缩 |
| 3 | 领域关键 | Skill中的领域规则章节 | **用户消息**（首条） | 不可压缩 |
| 4 | 可压缩 | 对话历史 + 中间结果 | 用户消息 | 可压缩 |

**拼装后的Prompt结构**：

```
[system prompt — 始终不变，可被LLM服务缓存]
你是一个科研助手。你必须遵守以下约束：
1. 严格忠于原始内容，不得歪曲、夸大或缩小
2. 不确定的信息不要编造，宁可不写也不要瞎写
3. 如实标注你的不确定之处
4. 优先保证准确性，其次才是完整性
{场景特定约束}

[user message — Skill + 业务输入作为首条用户消息注入]
以下是你在{场景}中应遵循的个性化知识和规则：
{SKILL.md全文}

{Pipeline提供的结构化数据}

[输出格式要求]
请严格按照以下JSON Schema返回结果：
{output_schema的描述}
```

### 2.3 ScenarioRegistry 场景注册表

所有LLM调用场景通过中央注册表管理，实现场景自文档化、前置条件自动校验、新增场景零改动Gateway核心代码。

**注册结构**：

```python
registry.register(
    name="paper_search",           # 场景标识
    mode="async",                  # 调用模式：async / stream
    requires_web_search=True,      # 是否需要联网搜索
    input_schema={...},            # 输入结构定义
    output_schema={...},           # 输出结构定义
    check_fn=lambda: bool(config.get("web_search_tool_name")),  # 前置条件检查
    requires=["web_search"],       # 依赖声明
)
```

**注册表职责**：

| 职责 | 说明 |
|------|------|
| 场景发现 | Pipeline通过场景名从注册表获取完整定义，无需硬编码 |
| 前置校验 | 调用前自动执行check_fn，不满足则提前报错而非等到LLM调用失败 |
| 依赖声明 | requires列表声明场景依赖（如web_search），Gateway据此决定是否注入联网搜索工具 |

**自动发现**：任何 `scenarios/*.py` 文件中包含顶层 `registry.register()` 调用即被自动导入，无需维护import列表。

### 2.4 重试与错误处理

LLM调用失败时自动重试（最多3次，间隔递增1s→2s→4s）。3次均失败则：异步模式写入operation_logs并显示失败状态；流式模式推送错误消息，用户可手动重试。

### 2.5 Token计量

每次LLM调用后，Gateway记录Token使用量：

```json
{
  "scenario": "paper_search",
  "model": "gpt-4o",
  "input_tokens": 3200,
  "output_tokens": 450,
  "timestamp": "2026-04-23T10:30:00Z"
}
```

前端在左下角展示累计Token用量，按功能模块分组。

## 3. Pipeline 设计

每个功能模块是一条确定性管道。管道中的步骤是固定的，LLM只出现在"需要理解/判断/生成"的环节。以下 Pipeline 中未标注"LLM Gateway调用"的步骤均为确定性操作（数据库读写、文件管理、WebSocket推送等），不再逐条标注"确定性"。

### 3.1 论文搜索Pipeline（异步）

**触发方式**：APScheduler每日定时触发

```
Scheduler定时触发
    │
    ▼
[Step 1] 读取论文库领域描述
    │  SQLite查询 → SELECT domain_description FROM paper_libraries WHERE id = ?
    ▼
[Step 2] LLM Gateway调用：paper_search
    │  输入：领域描述
    │  联网搜索：✅（传入web_search工具）
    │  LLM：搜索领域及强相关领域最新论文、筛选（错过好过泛滥）、生成结构化摘要（≤200字）
    │  约束：论文必须真实存在，必须是最新的
    │  迭代：若LLM判断结果不足，可发起补充搜索（最多3轮迭代，每轮结果累积）
    │  输出：筛选后的论文列表（含摘要）
    ▼
[Step 3] 解析LLM输出JSON → 去重 → 写入SQLite
    │  JSON解析
    │  去重规则：URL和标题均相同则判定重复，不插入
    │  INSERT INTO papers + paper_library（跳过重复论文）
    ▼
[Step 4] 通知前端刷新时间线
    │  WebSocket推送
```

**PaperItem统一结构**（LLM输出→内部格式）：

```json
{
  "title": "...",
  "authors": ["Author A", "Author B"],
  "institution": "MIT",
  "abstract": "...",
  "summary": "LLM生成的结构化摘要，≤200字",
  "published_date": "2026-04-20",
  "source_url": "https://arxiv.org/abs/2401.12345"
}
```

### 3.2 本地论文导入Pipeline（异步）

**触发方式**：用户通过Web GUI上传PDF

```
用户上传PDF
    │
    ▼
[Step 1] 存储PDF到论文库文件夹
    │  文件写入 data/paper_libraries/{库名}/papers/
    ▼
[Step 2] LLM Gateway调用：paper_metadata
    │  输入：PDF全文文本
    │  LLM：提取标题、作者、作者院校、摘要
    │  输出：{title, authors, institution, abstract}
    ▼
[Step 3] LLM Gateway调用：paper_search（仅摘要生成部分）
    │  输入：论文元数据 + 领域描述
    │  LLM：生成结构化摘要（≤200字，非一句话）
    │  输出：{summary: string}
    ▼
[Step 4] 写入SQLite + 标记来源为"手动导入"
    │  INSERT INTO papers
    │  去重规则：标题相同则判定重复，不插入
    ▼
[Step 5] 论文加入时间线
    │  WebSocket推送
```

### 3.3 深度阅读Pipeline（流式）

**触发方式**：用户点击论文的"深度阅读"按钮

```
用户点击"深度阅读"
    │
    ▼
[Step 1] 前置检查
    │  检查论文全文是否已下载（pdf_path非空）
    │  若未下载 → 提示用户先下载全文
    ▼
[Step 2] 加载论文全文 → 渲染到中间区域
    │  读取PDF文件，前端react-pdf渲染
    ▼
[Step 3] 进入对话循环 ───────────────────────────┐
    │                                              │
    ▼                                              │
[Step 4] 用户提问                                  │
    │                                              │
    ▼                                              │
[Step 5] LLM Gateway调用：deep_reading_chat        │
    │  输入：论文全文 + 对话历史 + 用户问题          │
    │  LLM：基于论文全文内容回答                     │
    │  输出：流式文本                                │
    │                                              │
    ▼                                              │
[Step 6] 用户继续提问或结束 ───────────────────────┘
    │
    ▼ (用户关闭论文或切换功能)
[Step 7] 用户手动确认是否生成总结
    │  若用户认为讨论有意义 → 确认生成
    │  若用户认为无意义 → 跳过，不生成总结
    ▼
[Step 8] LLM Gateway调用：deep_reading_summary
    │  输入：完整对话记录
    │  LLM：总结为≤1000字
    │  输出：{summary: string}
    │  注：同一篇论文只保留一份总结，每次确认时LLM基于全部对话记录重新生成，替换旧总结
    ▼
[Step 9] 写入SQLite
    │  UPDATE papers SET deep_reading_summary = ? WHERE id = ?
```

### 3.4 idea锤炼Pipeline（流式）

**触发方式**：用户提交idea

```
用户提交idea描述
    │
    ▼
[Step 1] 创建idea记录
    │  INSERT INTO ideas，状态="锤炼中"
    ▼
[Step 2] LLM Gateway调用：idea_analysis
    │  输入：idea描述 + Skill(idea锤炼)
    │  联网搜索：✅（LLM联网搜索相关论文进行新颖性评估）
    │  LLM：新颖性评估（基于联网搜索，不依赖本地论文库）、假设梳理、可行性分析、风险识别
    │  输出：{novelty, hypotheses, feasibility, risks}
    ▼
[Step 3] 进入多轮对话循环
    │  LLM Gateway调用：idea_chat（流式）
    │  输入：idea文档 + 对话历史 + 用户输入
    │  LLM：追问、建议、修正
    │  输出：流式文本
    │
    │  锤炼过程中，LLM联网搜索相关论文
    │  用户标记为"参考论文"的论文记录到idea_references
    ▼
[Step 4] 用户决定idea状态
    │  UPDATE ideas SET status = ? WHERE id = ?
    │  若"立项" → 触发任务创建流程
```

### 3.5 综述生成Pipeline（流式）

**定位**：研究文档演化的第二阶段——从 idea 出发，通过与 LLM 讨论，基于引用池论文撰写综述。

**触发方式**：用户在任务内请求综述讨论

```
用户请求综述讨论
    │
    ▼
[Step 1] 读取任务上下文
    │  读取idea文档 + 引用池论文列表
    ▼
[Step 2] 进入多轮对话循环 ───────────────────────────┐
    │                                              │
    ▼                                              │
[Step 3] LLM Gateway调用：review_discuss            │
    │  输入：idea文档 + 引用池论文 + 对话历史 + 用户输入
    │  LLM：基于引用池论文和idea中的研究方向，讨论综述结构、梳理领域脉络、撰写各部分内容
    │  约束：对每篇论文描述忠于原文
    │  输出：流式文本                                │
    │                                              │
    ▼                                              │
[Step 4] 用户继续讨论或结束 ─────────────────────────┘
    │
    ▼
[Step 5] 用户确认 → LLM生成综述部分
    │  LLM Gateway调用：review_generate
    │  输入：完整对话记录 + idea文档 + 引用池论文
    │  LLM：生成综述部分（Markdown），自动关联引用池中的BibTeX条目
    │  输出：流式Markdown文本
    ▼
[Step 6] 渲染综述到前端
    │  Markdown渲染
    ▼
[Step 7] 用户可要求迭代修改
    │  → 再次调用review_generate，带上已有综述+修改要求
    ▼
[Step 8] 综述部分写入研究文档
    │  文件写入 data/tasks/{task_id}/research.md
    │  研究文档从"idea"演化为"idea + 综述"
```

### 3.6 方法讨论Pipeline（流式）

**定位**：研究文档演化的第三阶段——基于综述中的已有工作，通过与 LLM 讨论，梳理方法框架。至此研究文档从粗糙想法完善为论文雏形。

**触发方式**：用户在任务内（综述完成后）请求方法讨论

```
用户请求方法讨论
    │
    ▼
[Step 1] 读取任务上下文
    │  读取研究文档（idea + 综述）+ 引用池论文列表
    ▼
[Step 2] 进入多轮对话循环 ───────────────────────────┐
    │                                              │
    ▼                                              │
[Step 3] LLM Gateway调用：method_discuss            │
    │  输入：研究文档（idea + 综述）+ 引用池论文 + 对话历史 + 用户输入
    │  LLM：基于综述中的已有工作和idea中的假设，讨论方法框架、技术路线、创新点表述
    │  输出：流式文本                                │
    │                                              │
    ▼                                              │
[Step 4] 用户继续讨论或结束 ─────────────────────────┘
    │
    ▼
[Step 5] 用户确认 → LLM生成方法部分
    │  LLM Gateway调用：method_generate
    │  输入：完整对话记录 + 研究文档（idea + 综述）
    │  LLM：生成方法部分（Markdown），包含方法框架、关键算法描述、与已有工作的区别
    │  输出：流式Markdown文本
    ▼
[Step 6] 渲染方法部分到前端
    │  Markdown渲染
    ▼
[Step 7] 用户可要求迭代修改
    │  → 再次调用method_generate，带上已有方法部分+修改要求
    ▼
[Step 8] 方法部分写入研究文档
    │  文件写入 data/tasks/{task_id}/research.md
    │  研究文档从"idea + 综述"演化为"idea + 综述 + 方法 = 论文雏形"
```

### 3.7 提示词文档生成Pipeline（流式）

**触发方式**：用户在任务内请求生成提示词文档

```
    │  LLM：追问缺失信息 + 生成提示词文档（确保实验设计与研究目标一致）

[Step 3] 用户可随时编辑文档内容
```

**触发方式**：用户上传训练日志

```
    │
    ▼
[Step 1] 存储日志文件到任务目录
    │  文件写入 data/tasks/{task_id}/experiments/
    ▼
[Step 2] LLM Gateway调用：experiment_analysis
    │  输入：训练日志JSON + Skill(代码提示词生成)
    │  LLM：分析收敛/敏感性/问题，给出改进建议
    │  输出：Markdown分析报告

[Step 3] 渲染分析报告到前端

```

### 3.9 论文润色Pipeline（流式）

**触发方式**：用户上传.docx文件提交润色请求

```
用户上传.docx文件
    │
    ▼
[Step 1] LLM Gateway调用：paper_polish
    │  输入：原文 + Skill(论文润色: 禁用词列表+术语映射表+风格偏好)
    │  LLM：润色 + 差异标注
    │  输出：{diffs: [{orig, modified, reason}]}
    │  约束：严格保持原文语义，不得改变论点/结论/数据含义
    ▼
[Step 2] 渲染内联差异标注
    │  前端根据diffs渲染
    ▼
[Step 3] 用户审阅
    │  逐条接受/拒绝，或一键全部接受/拒绝
    │  用户可输入自然语言指令要求再次润色
```

### 3.10 进化Pipeline（异步+用户确认）

**触发方式**：对话结束 或 操作日志积累

```
交互产生学习信号
    │
    ▼
[Step 1] LLM Gateway调用：evolution_observe
    │  输入：对话记录 或 操作日志片段
    │  LLM：提取可学习的模式
    │  输出：{observation, dimension}
    ▼
[Step 2] 写入 observing.md
    │  追加写入 skills/{Skill名}/observing.md
    ▼
[Step 3] LLM Gateway调用：evolution_pattern
    │  输入：observing.md全文
    │  LLM：判断是否存在语义重复的模式
    │  输出：{is_repeated, pattern, merged_observation}
    ▼
[Step 4] 若is_repeated=true
    │  写入candidate.md，从observing.md删除对应条目
    ▼
[Step 5] 提示用户确认
    │  前端弹出确认通知
    ▼
[Step 6] 用户确认 → LLM Gateway调用：skill_compress
    │  输入：当前SKILL.md全文 + 新知识
    │  LLM：合并新知识并压缩至≤800字
    │  输出：{compressed: string}
    ▼
[Step 7] 写入SKILL.md，从candidate.md删除对应条目
    │  文件写入
    ▼
[Step 8] 记录进化日志
    │  INSERT INTO evolution_logs
```

### 3.11 上下文压缩Pipeline（自动触发，所有LLM场景）

**触发方式**：LLM Gateway检测到上下文长度 ≥ 80%阈值

**适用范围**：所有LLM调用场景——无论是流式对话还是异步处理，只要上下文接近阈值，就触发压缩。

```
LLM Gateway检测到上下文长度 ≥ 80%
    │
    ▼
[Step 1] 锁定不可压缩层
    │  按层级标记
    │  不可压缩：系统级约束 + Skill全文 + Skill中的领域规则
    ▼
[Step 2] LLM Gateway调用：context_compress
    │  输入：可压缩层内容 + Skill中的领域规则
    │  LLM：根据领域规则标注优先级 + 执行压缩
    │  - 匹配领域规则中的核心概念/关键术语 → 保留原文
    │  - 与领域相关但未命中核心规则 → 压缩为摘要
    │  - 与领域无关 → 高度概括
    │  输出：{compressed_context: string}
    ▼
[Step 3] 替换可压缩层，保留不可压缩层
    │  上下文替换
    ▼
[Step 4] 通知前端"上下文已压缩"
    │  WebSocket推送轻量提示
```

## 4. 数据模型设计

### 4.1 SQLite核心表

```sql
CREATE TABLE paper_libraries (
    id                  INTEGER PRIMARY KEY AUTOINCREMENT,
    name                TEXT NOT NULL,
    domain_description  TEXT NOT NULL,
    created_at          TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE papers (
    id                  INTEGER PRIMARY KEY AUTOINCREMENT,
    title               TEXT NOT NULL,
    authors             TEXT,
    institution         TEXT,
    abstract            TEXT,
    structured_summary  TEXT,
    source              TEXT NOT NULL CHECK(source IN ('llm_search', 'manual')),
    published_date      TEXT,
    source_url          TEXT,
    pdf_path            TEXT,
    deep_reading_summary TEXT,
    created_at          TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE paper_library (
    paper_id            INTEGER NOT NULL REFERENCES papers(id),
    library_id          INTEGER NOT NULL REFERENCES paper_libraries(id),
    is_read             INTEGER NOT NULL DEFAULT 0,
    is_interested       INTEGER NOT NULL DEFAULT 0,
    PRIMARY KEY (paper_id, library_id)
);

CREATE TABLE ideas (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    title       TEXT NOT NULL,
    content     TEXT,
    status      TEXT NOT NULL DEFAULT '锤炼中' CHECK(status IN ('锤炼中', '已放弃', '已暂停', '已立项')),
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE idea_references (
    idea_id     INTEGER NOT NULL REFERENCES ideas(id),
    paper_id    INTEGER NOT NULL REFERENCES papers(id),
    PRIMARY KEY (idea_id, paper_id)
);

CREATE TABLE tasks (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    name            TEXT NOT NULL,
    research_goal   TEXT,
    source_idea_id  INTEGER REFERENCES ideas(id),
    created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE task_references (
    task_id     INTEGER NOT NULL REFERENCES tasks(id),
    paper_id    INTEGER NOT NULL REFERENCES papers(id),
    bibtex      TEXT,
    tags        TEXT,
    added_at    TEXT NOT NULL DEFAULT (datetime('now')),
    PRIMARY KEY (task_id, paper_id)
);

CREATE TABLE experiments (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id         INTEGER NOT NULL REFERENCES tasks(id),
    log_path        TEXT NOT NULL,
    analysis_report TEXT,
    created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE evolution_logs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    content     TEXT NOT NULL,
    source      TEXT,
    dimension   TEXT NOT NULL,
    level       INTEGER NOT NULL CHECK(level IN (1, 2, 3)),
    skill_name  TEXT NOT NULL,
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE operation_logs (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    operation_type  TEXT NOT NULL,
    operation_object TEXT,
    result          TEXT,
    timestamp       TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE llm_usage (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    scenario        TEXT NOT NULL,
    model           TEXT NOT NULL,
    input_tokens    INTEGER NOT NULL,
    output_tokens   INTEGER NOT NULL,
    timestamp       TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE conversations (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id     INTEGER REFERENCES tasks(id),
    scenario    TEXT NOT NULL,
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE conversation_messages (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    conversation_id INTEGER NOT NULL REFERENCES conversations(id),
    role            TEXT NOT NULL CHECK(role IN ('system', 'user', 'assistant', 'tool')),
    content         TEXT NOT NULL,
    timestamp       TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE paper_notes (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    paper_id    INTEGER NOT NULL REFERENCES papers(id),
    content     TEXT NOT NULL,
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

### 6.1 REST API

| 方法 | 路径 | 说明 |

### 6.2 WebSocket事件

| 事件方向 | 事件名 | 说明 |
|----------|--------|------|
| 服务端→客户端 | `llm.stream.chunk` | 流式LLM响应的文本chunk |
| 服务端→客户端 | `llm.stream.done` | 流式LLM响应完成 |
| 服务端→客户端 | `llm.stream.error` | 流式LLM响应出错 |
| 服务端→客户端 | `async.progress` | 异步任务进度（如12/30） |
| 服务端→客户端 | `async.complete` | 异步任务完成 |
| 服务端→客户端 | `async.failed` | 异步任务失败 |
| 服务端→客户端 | `context.compressed` | 上下文已压缩通知 |
| 服务端→客户端 | `conversation.restored` | 对话会话恢复通知（刷新/重启后自动恢复） |
| 服务端→客户端 | `evolution.candidate` | 进化候选知识待确认 |
| 客户端→服务端 | `chat.message` | 用户发送对话消息 |

## 7. 安全与隐私

| 措施 | 说明 |
|------|------|
| API Key 本地存储 | LLM API Key 仅存储在用户本地配置文件中，不上传任何服务器 |
| 论文数据本地存储 | 所有论文全文和索引数据存储在本地，不上传云端 |
| 离线可用 | 已下载论文的深度阅读、Skill 进化、引用管理、综述生成、方法讨论、论文润色、对话历史查看等均可离线使用 |
| LLM 数据流向 | 用户数据仅发送到用户自行配置的 LLM 服务端；选择国内大模型时，数据仅在国内模型服务端处理 |
